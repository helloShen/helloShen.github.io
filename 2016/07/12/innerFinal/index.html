为什么java匿名内部类的参数引用必须是final？

Posted on 2016-07-12   |   In Note   |     |     Viewed times

也是知乎上的回答，搬过来记录一下。

作者：胖胖
链接：https://www.zhihu.com/question/21395848/answer/110829597
来源：知乎
著作权归作者所有。商业转载请联系作者获得授权，非商业转载请注明出处。

先说结论，这不是我们的错，这是Java语言本身的问题。语言本身也有一个进化的过程，这个问题在Java 8里已经改进了，不显式声明final编译器不会报错了。但参数仍旧不能改，必须是effectively final。这点上说Java还是有是局限的。至少其他很多语言都没这个问题。

为什么会造成这个尴尬的局面呢？先引用StackOverFlow这个问题最高票答案的一句话，

When you create an instance of an anonymous inner class, any variables which are used within that class have their values copied in via the autogenerated constructor.

实际的情况是这样，因为java内部类保留一个指向外部类的引用，因此可以自由访问外部类的所有字段。所以程序员在编程的时候，会很自然地认为内部类能对外部类的字段做任意的操作或修改。这真的天经地义。如果让乔布斯来设计java那一定就是这样的。

但现实和我们的YY总是有那么一点差距，这个差距就在于：当一个数据被传输进内部类里面，成为内部类某个实例的 “local variables” 局部变量的时候，编译器会给这些数据拍一个”定妆照“，也就是做一份拷贝传到内部类。也就是那句”have their values copied in“的意思。

那哪些变量会传进内部类里成为局部变量呢？那就是负责返回匿名内部类引用的那个方法的参数了。这货平时不会发作，但一旦我们把外部类的某字段当参数传进去的时候，就出问题了。大多数程序员还会想当然地以为内部类和外部类是通的，所以在内部类里修改这个参数值，能和外部类保持同步。但其实大家都被骗了，因为外部类字段的引用没有传进去，传进去的只是它的一份拷贝，影分身。

但这样的bug编译器本身是不报错的，很隐蔽。所以java索性就硬性规定传进去的值不许改，好粗暴呀。。啊呀呀

说了这么多，还是来看一下代码，代码比较直观好理解。

代码其实很简单，

  • 外部类的唯一字段：info
  • 外部类有一个返回匿名内部类引用的方法getAnonymousInner( String paramInfo)
  • 我故意把外部类的字段info当参数传给getAnonymousInner来创建内部类实例
  • 内部类中的方法负责打印传进去的参数
  • 外部类的changeInfo(String newInfo)方法修改外部类字段
  • 再次让内部类打印参数

实验结果如下，

内部类实例创建的时候，和外部类字段是一致的。但外部字段被修改之后，内部类传进去的参数并没有跟着外部类字段同步。

这就是故事的全部。如果实在想把内部类的结果传出来的话，就只好手动boxing一下了，比如最常用的包装到一个String[1]里面。因为java的array是一个引用，所以我们可以改array指针指向的那个值，但array本身储存的地址是不变的。如此曲线救国。

如果想知道其他语言，比如c++是怎么解决这个问题的，可以去看 @崔鹏飞 的《为什么必须是final的呢？》。作者在里面向我们展示了反向编译的结果，看得出其他没有这个问题的语言，都是把这个参数自动boxing一下再传进去的，所以能做到自动同步。
