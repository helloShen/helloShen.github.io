Think in Java 读书笔记：第十一章 - 容器

Posted on 2016-07-10   |   In Note   |     Viewed times

摘要

容器，容器，容器
我对容器感兴趣的一个重要原因是，我对java.util库的内部实现感兴趣。创造Java的一群人写的java代码，是学习java少有的靠谱的捷径。

Array不是一般意义上的“容器”

讲容器之前，请允许我先把Array和其他所有的容器区分开来。它是Java语言内置(compiler-supported type)的数据类型，它是唯一可以存放基本型的容器。而且Array非常地高效。单单从效率的角度比较，Array明显优于List。
这里有一篇好文章：《历史上的 Collection 类 ― 数组》，可以帮助我们了解Array的基本情况。

Array为什么效率高呢？这要从Array本身在内存中的结构说起（下图）：

对于Array的具体实现，有下面几个重要事实：

  • Array是一个对象Object
  • Array在Stack栈区只占一个“long”型，储存指向在Heap堆区的实际数据首地址的引用
  • Array在Heap堆区的具体数据，所占的是“连续空间”


如上图，多维数组的实现方式，是靠一个“引用的数组”来过度的。第一层数组的每一个地址，只储存指向具体数据数组的一个引用。

Array效率高的秘密，就在于它在内存中占据的空间是连续的。因此如下图，只要掌握了在内存中的首地址，以及数组的长度，对于任意想要的元素，数组只要在首地址基础上加上偏移值，就能找到该元素。而对内存的操作，seek偏移值的开销是非常小的。

但连续空间是把双刃剑，副作用是Array声明的时候必须固定长度，之后不能修改。这极大地限制了Array的使用，因为大多数的使用场景，声明数据的时候，并不知道我们需要多大的容器。这才导致了后面其他容器的出现。

但如果对性能敏感的时候，如果可以固定大小，还是首选Array。尤其是储存基本型的时候。

java.util类库中的容器

先声明，这一章只负责告诉读者一些关于util类库定义的容器的一些有用的结论，来帮助我们在编程的时候尽量选择正确合适的容器。至于容器内部的一些算法细节，会在之后的《容器深谈》这一章中探讨。

Java类库java.util提供的容器到底有哪些？让我们看着下面的这张图，整理一下。

  • 容器按功能主要分成两大分支：Collection和Map。各自以Interface接口的形式存在。
    Collection指的是一组元素的集合。强调的是元素本身。 Map的特殊功用在于它为每个元素维护的一组“KEY-VALUE”对。Key用来搜索value值。典型的应用就是“词典”。

顾名思义，Collection指的是一组元素的集合。强调的是元素本身。它手下有耳熟能详的三大将：

  • List
  • Set
  • Queue
    他们都有各自的强项。

它们也都是Interface接口。最正统的是List，强调“一组元素的序列”。通常是元素插入集合时候的顺序。

Set的特点是其中元素的唯一性。插入新元素前会检查已有列表，如果已经有了，就不会再插入。

Queue虽然也强调的是序列，但更主要的是“获取下一个元素的顺序”。最简单的一种就是“FIFO(First In First Out)”先进先出。

主力容器：ArrayList（动态数组）

为了弥补Array功能的限制，Java的util库实现了一系列的动态容器。其中最常用的应该是ArrayList（动态数组）。

对ArrayList最简单的描述：一个可以动态扩展的无限长度的数组。它就是为了解决Array固定长度的遗憾。

an array that automatically expands itself.

但鱼和熊掌不可兼得。ArrayList实现动态扩展数组长度，必然就要牺牲效率，为它的弹性付出代价。

声明ArrayList的时候，不规定类型的话，它接受任何Object。

在尖括号里声明类型的话，就只接受这种特定类型的数据。
